# GUI ↔ Headless ViewModel Test Pairing Audit

> Date: 2026-05-02
> Branch: codetracer-viewmodel
> Mission goal: #2 — "Each GUI test must have a corresponding headless ViewModel test"
> Scope: inventory-only. No new VM tests are written by this iteration; the
> goal is to surface the gap surface so future iterations can target specific
> clusters.

## 1. Methodology

### 1.1 Discovery

Specs and VM tests were discovered by glob (no execution):

```
find src/tests/gui/tests -name "*.spec.ts"   # GUI specs (Playwright)
find src/tests/gui/tests -name "*_test.nim"  # Headless VM tests (Nim unittest)
```

Per-spec test counts come from a `^\s*test\(` Grep across each `.spec.ts`.
Per-suite mapping in `isonim_views_test.nim` comes from a `^suite "` grep.

### 1.2 What counts as a "matching VM test"

Per `codetracer-specs/Testing/Test-Co-Location-Convention.md` (rule 1) and
`Testing-Guidelines.md` (Headless-First Policy):

A spec `tests/<topic>/<name>.spec.ts` has a matching VM test when there
exists a Nim test that:

  (a) Lives in the **same directory** OR exercises the same panel via the
      shared `views/isonim_views_test.nim` (the de-facto cross-panel
      view-rendering test file, despite the convention suggesting one
      `<topic>.vm-test.nim` per topic).
  (b) Drives the same VM/store signal flow via `MockBackendService` or the
      VM constructors (`createCalltraceVM`, `createStateVM`, etc.).
  (c) Covers at least one of the spec's test names by intent — either the
      same trigger (action, click, scroll, search input) or the same
      observable result (selection signal, visible row count, dispatched
      DAP command).

### 1.3 Confidence labels

* **PAIRED** — same directory + name overlap + same flow.  High confidence
    the VM test reproduces the GUI test's failure shape.  One-to-one or
    near-one-to-one with the spec.
* **PARTIAL** — VM coverage exists for the panel but doesn't exercise the
    same flow / scenario as the spec (e.g. structure-only, missing the
    user action the GUI spec triggers).
* **GAP** — no VM test for this spec's panel/flow.  A GUI failure here
    cannot be reproduced at the headless layer.
* **SHARED** — multiple specs are covered by a single broad VM test
    (typically `views/isonim_views_test.nim` or
    `scenarios/feature_scenarios_test.nim`).  Coverage exists but is not
    co-located per the convention.

### 1.4 Limits of this audit

* No tests are run; coverage is judged from file names, suite/test titles
    and reading commit messages of the migration commits (1.1–1.51).
* Some specs have many sub-tests; only the spec as a whole is rated.
* The mapping is a starting map for follow-up iterations, not a final
    unit-by-unit audit.

## 3. Gap clusters

GAP specs grouped by panel/area, with effort estimate and leverage
assessment for follow-up iterations.

### 3.1 Cluster A — Per-language `*_example.spec.ts` smoke (12 GAPs, ~110 tests) — CLOSED at smoke level (1.63, 2026-05-02)

**Specs:** aiken, cadence, cairo, circom, leo, masm, move, polkavm,
solana, stylus, sway, tolk.

**Common shape:** each spec has 6–10 tests checking editor-loads,
event-log-populated, calltrace-navigation, locals-populated,
flow-renders, output-contains-X.  Same shape as the SHARED sudoku
cluster.

**Effort:** **small per language**.  Pattern is mechanical: extend
`integration/language_smoke_test.nim` (or split into per-language
`<lang>-example.vm-test.nim` siblings) with a single trace fixture per
recorder.  All 12 recorders are already audited.

**Leverage:** ★★★★★ — closes 12 spec rows, ~110 GUI test scenarios with
one extension to the shared smoke harness.

**Closure (1.63, 2026-05-02):** added
`integration/language_smoke_mock_test.nim` with 12 Mock-driven smoke
suites — one per language — that drive a `MockBackendService`-fronted
`ReplayDataStore` and assert on the four observable signal flows the
specs exercise (editor settles on `.<ext>` source file via
`activeFileName` memo, calltrace populates with the language's expected
entry function via `visibleLines` memo, state panel exposes locals via
`currentVariables` memo, event log surfaces a row via `eventRows`
signal).  All 12 pass on both `just test-vm-native` and
`just test-vm-js`.  The existing real-backend
`language_smoke_test.nim` is intentionally untouched — it is excluded
from the VM harnesses (see the `! -path '*/integration/language_smoke_test.nim'`
filter in the justfile) because it spawns a real `replay-server`.
The new file is the Mock complement and lives alongside it in
`integration/`.  Cluster A specs move from GAP → PAIRED at the smoke
level; per-language deeper assertions (e.g. specific event count,
exact stack-frame contents on masm, ecalli register staging on
polkavm) remain as follow-ups under §3 / §5 of this audit and as the
next-iteration cluster A.1 in §5.5 of the migration doc.

## 5. Recommendations

### 5.1 Co-location convention compliance

The convention in `Test-Co-Location-Convention.md` requires
`<topic>.vm-test.nim` siblings of `<topic>.spec.ts`.  Observed
violations:

* **All 13 sudoku specs** lack co-located VM siblings; coverage
    is centralised in `scenarios/feature_scenarios_test.nim` and
    `views/isonim_views_test.nim`.  Either accept the centralisation
    (and amend the convention to allow shared-language smoke files) or
    split per-language.
* **All 17 program_specific specs** lack co-located VM siblings.
* **build/** specs are paired only via `views/isonim_views_test.nim`,
    not via co-located `build/build-panel.vm-test.nim`.
* **VM test naming uses `_test.nim`** (e.g. `calltrace_vm_test.nim`)
    rather than the convention's prescribed `.vm-test.nim` (e.g.
    `calltrace-vm.vm-test.nim`).  The convention document itself notes
    its status as "Proposed — Directory restructuring pending".

**Recommendation:** the convention document should be **amended** to
acknowledge that:

  (a) Cross-panel rendering tests (`views/isonim_views_test.nim`)
      legitimately cover many `<topic>.spec.ts` files at once.  The
      "one .vm-test.nim per .spec.ts" rule is too strict.
  (b) Per-language smoke is a well-defined family that legitimately
      lives in a shared file (`integration/language_smoke_test.nim`).
  (c) The actual filename pattern in the repo is `<topic>_<vm>_test.nim`
      (e.g. `calltrace_vm_test.nim`); update the convention to reflect
      this or schedule a `git mv` migration.

### 5.2 Tooling gap — no CI check

No CI script enforces the spec ↔ VM-test pairing.  The convention
document proposes a check but it is not wired up:

```bash
for spec in tests/**/*.spec.ts; do
  vm_test="${spec%.spec.ts}.vm-test.nim"
  [ -f "$vm_test" ] || echo "MISSING: $vm_test"
done
```

**Recommendation:** add a soft check (warning, not failure) to CI that
flags new specs landing without a VM sibling.  Until the convention is
ratified to allow shared files, the check should accept either a
co-located sibling OR an entry in a small mapping table (e.g.
`tests/.spec-vm-mapping.json`) pointing at a shared VM test file.
This is the lowest-friction way to prevent coverage regression.
